# DOCLFE — Dsls Open Core License (Free Edition)

> 一个源码强制公开程度高于 GPLv3 的免费许可证框架。不属于 OSI 批准的开源许可证。

**DOCLFE（Dsls Open Core License Free Edition）v1.0** 是 DOCL 许可证家族的非商业变体，由 Dsls 团队设计。
所有修改和衍生作品必须公开源码，强制程度高于 GPLv3（含反 TiVo 条款、AGPL 级的联网触发公开义务），
但**不涉及商业授权**，可免费用于任何用途。

本仓库同时托管 **DOCL v1.0 标准版（Standard Edition）**（含商业授权条款），详见下文。

---

## 为什么需要 DOCL？

现有的许可证模型各有各的局限：

| 许可证 | 问题 |
|--------|------|
| **GPL/AGPL** | 强制开源很强，但商业模式为零。项目方没法靠许可证本身盈利 |
| **BSL** | 有 Change Date，商业限制是临时的；定价权完全在项目方，社区无话语权 |
| **ELv2 / RSAL** | 只防云厂商，不防其他商业行为；定价不透明，社区无参与 |
| **MIT / Apache 2.0** | 完全开放，但项目方被大公司白嫖，无法持续维护 |

**DOCL 的答案：**

许可证本身就是一个治理契约——它不只是一个法律文本，还是一个项目方和社区之间的 **权力分配协议**。

## 三层架构

DOCL 不是一个单一的许可证，而是一个 **许可证体系**：

| 层级 | 性质 | 内容 |
|------|------|------|
| **① 主许可证** DOCL v1.0 | **不可变更** | 源码强制公开 · 反竞争性分支 · CLA + 回采权 · 商业使用需授权 · 免责 |
| **② 额外条款** | **可自定义** | 项目方自行填写豁免场景、技术保护措施等，不得与主许可证冲突 |
| **③ 定价与社区治理** | **不可变更** | 初始定价 $100k/月 · 调价需社区 51% 投票 · ≥15% 须公示 · 存量用户保护 |

## 核心特征

### 源码强制公开（参考 GPLv3 标准）

任何对本软件的修改或衍生作品，无论是内部使用还是外部分发，无论商业还是非商业，必须完整公开其对应源码（Corresponding Source）。公开方式包括物理介质、书面要约、网络分发等五种途径。

### 反 TiVo 化保护

如果软件在用户产品（消费电子设备等）中分发，必须附带安装信息，确保修改后的版本可以在该设备上正常运行——堵住了"给源码但不让你装回去"的漏洞。

### 反竞争性分支

以付费方式向第三方提供功能显著重叠的产品（包括托管服务、嵌入形式或付费支持），且功能未实现显著分化的分支，许可证自动终止（参考 HashiCorp BSL 模式）。内部使用、非付费分发及首次发布时不具竞争性的产品不受影响。

### CLA + 贡献回采

所有贡献者须签署 CLA，贡献版权归项目方。项目方保留从任何分支中回采代码的权利，防止社区改进被竞品锁死。

### 商业定价 + 社区治理

商业使用需购买授权，初始定价公开透明（$100,000/月）。任何调价须经社区 51% 投票通过，涨幅 ≥15% 须提前 15 天公示。调价不影响存量用户。

### 额外条款自由填写

版权持有方可根据项目需要填写额外条款，只要不违法且不与核心条款冲突即可。定价治理、豁免场景、技术保护措施等均可在此定义。

## 如何在自己的项目中使用 DOCL

将许可证文本复制到项目根目录的 `LICENSE` 文件中。须保持文本完整，不得仅通过 URL 链接引用。

同时准备以下文件（可选，按需提供）：
- **`ADDITIONAL_TERMS.md`** — 第八条（额外条款）的具体内容，包括你的项目特定的定价治理、豁免条件等；
- **`COMMERCIAL_EXEMPTION.md`** — 商业豁免条款的价格、条件及签约方式；
- **`CLA.md`** — 贡献者许可协议模板，或直接指向你的 CLA URL。

> ⚠️ **免责声明**：DOCL 是一个许可证框架，不构成法律建议。使用前请咨询法律专业人士，确保其符合你的项目需求和适用法律。

## 设计哲学

DOCL 的设计基于以下信念：

1. **开放是前提，不是施舍**——社区改进应当回馈给整个生态，而不是被私有化。源码强制公开是对所有用户的公平。
2. **可持续的商业模式是长期开放的保障**——没有收入的项目最终要么死掉，要么暗中收割用户。透明的商业化比隐藏的商业模式更健康。
3. **信任需要制度，不是承诺**——定价权交给社区，不是出于理想主义，而是因为这比"项目方承诺不涨价"更值得信任。
4. **许可证是契约，不是口号**——每一方的权利和义务都应当在许可证中写清楚，而不是留在"开源精神"的模糊地带。

## 许可证治理说明

本仓库采用 **DOCLFE** 作为项目许可证。以下为治理规则要点：

### 固定文本

许可证以项目中附带的固定文本为准，不得通过 URL 链接引用。您分发每一份复制件时，须附带完整文本。

### 主仓库升级权

版权持有方有权升级许可证版本，无需经社区同意。升级操作：
- 在仓库中创建一个独立提交（commit）
- 替换许可证文件并标注新版本号（如 v1.1）
- 提交后通过官方渠道公告

升级后，基于本项目的所有衍生仓库（包括但不限于 GitHub fork、git clone 后新建的仓库、镜像仓库）必须跟随，不得自行选择或固定版本。

### 许可证版本号

本仓库推送至 GitHub 的每个提交所附带的许可证文本，即为该版本所适用的条款。如许可证文件发生变更，新版本号应明确标注。

### 衍生仓库责任

基于本项目的任何衍生仓库，其许可证以主仓库当前版本为准。衍生仓库创建者不得自行选择或固定旧版本，否则视为竞争性分支，许可证自动终止。

### 模板文件

本仓库提供以下模板供项目使用：
- `CLA-template.md` — 贡献者许可协议模板
- `Commercital-Exemption-template.md` — 商业豁免协议模板
