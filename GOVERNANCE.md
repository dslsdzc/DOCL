# DOCL 许可证版本变更流程与独立提交规范

本文档说明 DOCL 主仓库如何进行许可证版本的主动升级操作。

---

## 1. 版本号规则

许可证版本号采用语义化方式：

```
v<主版本>.<次版本>

v1.0   ← 初始版本
v1.1   ← 漏洞修补 / 条款澄清
v2.0   ← 结构性变更 / 新增实质性义务
```

| 变更类型 | 版本号变动 | 示例 |
|---------|-----------|------|
| 漏洞修补、条款澄清、非实质性补充 | 次版本 +1 | v1.0 → v1.1 |
| 新增实质性义务、结构性重编 | 主版本 +1 | v1.x → v2.0 |

## 2. 独立提交规范

许可证升级必须通过一个**独立提交**完成，该提交不得包含任何代码变更。

### 提交内容

该提交应仅修改以下文件：
- 许可证文本文件（如 `DOCLFE-zh.txt`、`DOCL-en.txt`）
- 版本声明文件（如 `VERSION` 或许可证文件头部的版本号）

不得在该提交中混入：
- 代码改动
- 文档改动（与许可证无关的部分）
- 配置改动
- 其他任何与许可证无关的文件变更

### 提交信息格式

提交标题应使用以下格式：

```
license: upgrade to v<新版本号>

变更说明：
- <变更 1>
- <变更 2>
```

示例：

```
license: upgrade to v1.1

变更说明：
- 新增第 1.2(d) 条：AI/ML 模型训练属于修改
- 新增第 12.3a 条：主仓库升级权与衍生仓库跟随
```

### 文件变更示例

**版本号标注位置**：许可证文件头部或元数据区域应有版本号字段，每次升级时更新。

```
# DOCLFE v1.1 — Dsls Open Core License Free Edition
                          ↑
                      更新此版本号
```

## 3. 提交流程

```
步骤 1：创建独立分支
  git checkout -b license/upgrade-v1.1

步骤 2：修改许可证文件
  • 更新版本号
  • 应用条款变更
  • 核对所有交叉引用

步骤 3：检查提交内容
  git diff --stat
  → 确认只包含许可证文件变更

步骤 4：提交
  git add <许可证文件>
  git commit -m "license: upgrade to v1.1"
  git commit 信息按第 2 节格式

步骤 5：创建 Pull Request
  • PR 标题与 commit 标题一致
  • PR 描述列出变更摘要
  • 该 PR 不得混入代码变更

步骤 6：合并与公告
  • 合并至主分支
  • 通过 GitHub Release 或官方渠道发布公告
  • 公告内容包含变更摘要及生效说明
```

## 4. 升级后的强制跟随

许可证升级后，基于本项目创建的所有衍生仓库（包括但不限于 GitHub fork、git clone 后新建的仓库、镜像仓库及其进一步衍生）**必须跟随**，其许可证以主仓库当前版本为准。

衍生仓库创建者不得：
- 自行选择停留在旧版本
- 自行固定某一版本号
- 以未拉取升级 commit 为由拒绝跟随

违反上述规则的衍生仓库视为竞争性分支，其许可证自动终止。

## 5. 旧版本状态

主仓库升级后，旧版本许可证**不再对外授权**。但升级前已分发的复制件继续适用旧版本。升级后继续使用（包括运行、修改、分发）该软件的行为适用新版本。

## 6. 版本号与 Git 标签

建议在独立提交后为该提交创建 Git 标签，标签名与版本号一致：

```
git tag v1.1
git push origin v1.1
```

标签可作为版本的稳定引用点，也便于用户回溯。
